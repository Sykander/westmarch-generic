#### Factions

Factions are planned for a version higher than `1.0.0`. They are tracked here until the final target version is decided.

Target project:

- [factions project](../projects/factions/)

Expected scope:

- config-owned faction catalogue;
- `subsystems.factions` disabled by default;
- `!faction` player command hub;
- character-local membership and standing state;
- rank or renown support;
- shared faction requirement helpers;
- editor validation and guided editing;
- one or two representative integrations, such as faction-gated shops, quests, libraries, or location services.

#### Combat handoff and location maps

Exploration already has combat encounter rows and travel can emit `combat_add_prompt` follow-up text. The next-major plan is to turn that into a first-class combat handoff without making Avrae initiative automation mandatory.

Expected scope:

- add map metadata to configured locations, likely as `location.maps` or a referenced map catalogue;
- let encounter rows or pending encounter packages choose a map by id, tag, or explicit URL;
- fall back from encounter-specific maps to location defaults for `enc`, `hunt`, and travel-generated combat;
- include selected map or scene information in combat embeds and copy-paste handoff output;
- add a `!combat` command behind a new command toggle, with final placement decided before implementation (`misc.commands.combat` vs a small dedicated combat subsystem);
- support at least `!combat start`, `!combat preview`, and `!combat clear` if playtesting confirms GM control is needed;
- obey the existing `subsystems.travel.config.combat_add_prompt` modes: `madd_commands`, `combat_hint`, and `off`;
- preserve a copy-paste mode for servers where direct initiative mutation is unreliable or undesirable;
- update the editor model, validation, Check page, public docs, and `.alias-test` coverage for maps, pending encounters, and combat handoff.

Open design points:

- whether location maps should be inline location fields, a `world_data.maps_gvar_id`, or both;
- whether `!combat` belongs under `misc`, a future `combat` subsystem, or an existing exploration command hub;
- how much of Avrae initiative setup can be automated consistently across server permission setups;
- what the minimum pending encounter state should be so `!combat` can bridge `!hunt`, `!enc`, travel steps, and future dungeon hooks.

#### Economy progression commands

The existing `1.0.0` economy surface is `job`, `buy`, `sell`, and `wallet`. Later economy work should add progression commands around location work and player-owned production without hard-coding setting data into aliases.

##### `!interview`

Purpose: let characters acquire location jobs before those jobs become available through `!job`.

Expected scope:

- search configured jobs with the standard `lists.search_list` zero / one / many-result behavior;
- respect `location.commands.job`, job location restrictions, and `job_location_policy`;
- store character-local job eligibility, offers, or reputation state;
- let `!job` prefer or require acquired jobs when the server enables that policy;
- expose owner config for interview checks, DCs, cooldowns, failure handling, and visible job requirements.

##### `!business`

Purpose: let characters start and operate location-based businesses that can earn income and buy or sell goods.

Expected scope:

- configure business types, startup costs, operating costs, allowed locations, skills, and output tables;
- store character-owned or party-owned business state;
- run periodic earnings or loss checks using cooldowns, downtime, or world-clock policy;
- integrate with wallets, gp, shops, stock entries, and item outputs where configured;
- keep `!buy` and `!sell` as transactional commands while `!business` owns management and income rolls.

##### `!farm` / `!farming`

Working command name: `!farm`.

There is no existing implemented farming alias in this repo. Current gathering commands are `!forage`, `!fish`, `!mine`, `!lumber`, and `!hunt`; farming should be planned as a separate economy or downtime-production command, not as a rename of `!forage`.

Expected scope:

- configure land plots, crops, livestock, seasons, growth timers, upkeep costs, and harvest outputs;
- support location gating through current character location and configured farmable locations;
- use world time, downtime, or cooldown policy for growth and harvest pacing;
- write harvested goods to bags, wallets, shops, or business inventory according to server policy;
- decide before publication whether `!farming` should be an alias for `!farm` or remain only a docs/search keyword.

### Other Changes

#### Compatibility and migration rules

For `2.0.0`:

- identify every breaking command or config change in the version-specific release doc;
- provide `Upgrade Steps` for server owners, config maintainers, and developers adopting the release;
- preserve `1.x` behavior where practical with additive aliases or deprecation messages;
- make any required config migrations explicit and editor-checkable;
- keep new map, job acquisition, business, farm, and faction state config-owned and character/server-state-backed, not hard-coded in aliases;
- keep setting examples, such as Harpers data, in presets or owner config rather than generic aliases.
