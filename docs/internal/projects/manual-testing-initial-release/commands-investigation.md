# Commands Investigation

Source: `commands-test.md`, based on manual testing against config `5fd499a7-c865-43d2-941c-d32ee7975fb3` on the westmarch generic dev Discord server.

No code or test runs were performed for this investigation.

## Issue 1: `!help enc -here` Shows Placeholder Help

Observed output includes: `Example help doc for the !enc command.`

Investigation: `src/aliases/exploration/enc.md` is still a placeholder help document. The command runtime help from `!enc help` is generated by `exploration.help_text`, but Avrae `!help enc -here` uses the sourcemapped docs file.

What went wrong: implementation help and Avrae help docs diverged. The `.md` docs were not updated when `!enc` became usable.

Best fix:

- Replace `src/aliases/exploration/enc.md` with concise player-facing help.
- Start with what the command does and the basic syntax: `!enc <biome> [bonuses]`.
- Mention `!enc help` for server-aware runtime help.
- Include the config path at the bottom: `Configured under: Exploration -> enc`.
- Remove the placeholder heading/copy.

Same issue elsewhere: this is broad, not isolated to `enc`. Placeholder `Example help doc` text remains in many alias docs, including exploration, economy, content, misc, travel, and admin docs. Those should get the same pass so Avrae `!help` feels consistent.

Suggested content pattern:

- One-line purpose.
- Usage.
- Important arguments.
- Server setup note or config path.
- One or two real examples only when they work with the command's actual catalogue.

## Issue 2: Missing Encounter Text Loses Location Context

Observed output: `No encounters configured for forest / mine on this server.`

Investigation: `src/gvars/utils/exploration/exploration.gvar` calls `biomes.resolve_biome(...)`, receives only `(biome, error)`, then reports the generic missing-pool message when `encounter_lists.get_encounter_pick(...)` returns `None`.

The location resolver in `src/gvars/utils/world/biomes.gvar` can infer the biome from the character location, but it does not return the source location or whether the biome was manually supplied.

What went wrong: biome resolution collapses important context too early. By the time the missing-encounter message is built, the exploration runner cannot tell whether the player typed `forest` or whether the system inferred `forest` from `River Town`.

Best fix:

- Add a richer resolver, for example `resolve_biome_context`, that returns biome code, source mode, location id/name, and error.
- Use the location name in the missing-pool message when source mode is location.
- Keep the current text for manually supplied biome arguments.

Example target copy:

- Manual biome: `No encounters configured for forest / mine on this server.`
- Inferred biome: `No encounters configured for River Town / forest / mine on this server.`

Same issue elsewhere: all shared exploration activity commands that use `exploration.run_activity` can hit the same generic message: `enc`, `forage`, `fish`, `mine`, and `lumber`.

## Issue 3: `time` And `weather` Are Exposed But Not Implemented

Investigation: both `src/aliases/travel/time.alias` and `src/aliases/travel/weather.alias` are literal stubs that return `time is not implemented` or `weather is not implemented`. The backing `src/gvars/utils/world/clock.gvar` and `src/gvars/utils/world/weather.gvar` are placeholders.

What went wrong: these commands are present in the auth command map, defaults, editor subsystem toggles, and help docs even though the alias bodies are not implemented. The editor does warn when the commands are enabled, but the command can still be toggled on and invoked.

Best fix:

- Short-term: make the editor treat `time` and `weather` as planned/disabled controls, or make validation an error rather than a warning when they are enabled.
- Runtime: return a more helpful setup/planned message if invoked.
- Long-term: implement the clock/weather modules and aliases according to the existing design docs.

Same issue elsewhere: bare `!westmarch` HUD can include time/weather fields when those commands are enabled. The HUD has fallback logic, but the command implementations themselves remain stubs.

## Issue 4: `!buy Dagger` Gives A Lookup Failure With No Setup Hint

Observed output: `Found no shop stock matches for Dagger.`

Investigation: `src/aliases/economy/buy.alias` resolves the player's current location, asks `shops.find_shop_for_item`, and reports the returned lookup error. `src/gvars/utils/economy/shops.gvar` expects configured shops with explicit `stock` rows. It can read top-level `shops` or `world_data.shops`, but the starter config only shows shops as commented examples.

What likely happened: no shop at the character's location stocked a row matching `Dagger`, or no shops were configured/visible there. The error message does not distinguish those cases.

Best fix:

- In `shops.find_shop_for_item`, distinguish:
  - no shops configured at all,
  - shops exist but none at this location,
  - shops exist here but have no stock,
  - stock exists but the item query does not match.
- Add a setup hint when there are no shops or no stock rows.
- Consider showing available local shop names or a short stock preview when no stock matches.

Same issue elsewhere: `sell` uses the same shop/stock resolver and can produce similarly unhelpful setup failures.

Editor/docs follow-up: the editor has no guided shop/currency setup UI, so owners must hand-edit `shops` and `currencies` in raw JSON. That makes this command easy to enable before its required data exists.

## Issue 5: `!sell` Fails Without Making Missing Setup Obvious

Investigation: `src/aliases/economy/sell.alias` mirrors `buy` but requires shops that accept sells. The helper filters shops through `shops.list_shops(..., require_sells=True)` when finding an item. If no configured buying shop is visible, the user gets a generic stock/shop failure rather than a setup checklist.

What went wrong: the command is technically doing the configured-shop lookup, but the UX does not help a server owner or player understand which required config is absent.

Best fix:

- Reuse the improved shop diagnostics from the buy fix.
- For sell specifically, mention when shops exist but none have `accepts_sells: True`.
- If a shop is specified explicitly and does not buy, keep the current direct rejection but add the shop name.

Same issue elsewhere: `buy`, `sell`, and any future shop-backed economy command should share one setup-aware error helper.

## Issue 7: Travel Route Displays `Run !enc ` When No Biome Is Set

Observed output has command blocks like `Run !enc ` with a trailing space.

Investigation: `src/gvars/utils/world/paths.gvar` builds encounter-step text as `Run `{prefix}{activity} {biome}``. If `biome` is `None` or empty, the command string still includes the space. The editor path builder also creates new path/step rows with `biome: ''`, and validation currently checks path endpoints but not encounter step completeness.

What went wrong: runtime rendering assumes every encounter step has a biome, while the editor can export empty encounter steps.

Best fix:

- Runtime: only include the biome argument when it is non-empty.
- Runtime: if an encounter step has no biome and the server uses location-inferred exploration, display `Run !enc`; otherwise display a setup warning in `!travel` output.
- Editor: require a biome for encounter steps unless the intended route mode is explicitly location-inferred.
- Validation: warn or error for `world_data.paths.*.steps.*` encounter steps with no biome/activity where that would render an unusable command.

Same issue elsewhere: active journey display and `travel next` both use journey/path display helpers, so the formatting fix belongs in `paths.gvar`, not just `travel.alias`.

## Issue 8: Travel Route Steps Need Transport Icons/Config

Investigation: travel route planning supports transport modes (`walk`, `horse`, `boat`) through flags and path transport requirements, but display currently prints plain step bullets without transport icons. `src/gvars/configs/starter.gvar` includes a commented `world_data.transport` example, but runtime display does not use a transport emoji map.

Best fix:

- Add a transport display config, probably under `subsystems.travel.config.transport_icons` or `world_data.transport`.
- Defaults should include the requested meanings: walk, fly, horse, boat.
- Use those icons in `paths.display_path` or `journeys.display_journey` for each step/route.
- Add editor controls with a small default picker.

Same issue elsewhere: route preview, tracked journey display, and active journey text all render through `paths.display_path` / `journeys.display_journey`, so one shared display helper should own this.

## Issue 9: `scribe` Needs An Ignore/BYPASS Flag For Known Spell And Spell Slot Checks

Observed output: `No level 2 spell slots available.`

Investigation: `src/aliases/crafting/scribe.alias` passes parsed args into `crafting.run_command`, but `src/gvars/utils/crafting/crafting.gvar` has no `-i` handling. Unknown flags are stripped out of the subject by `subject_from_args`, but `build_plan` still enforces `require_known_spell`, and `_resource_checks` still checks spell slots when the spell-slot resource mode requires it.

What went wrong: the command has a legitimate 5e/Avrae use case for spells known through feats/features without slots, but the implementation only supports the normal RAW automation path.

Best fix:

- Recognize `-i` in `scribe`.
- When `-i` is present, bypass known-spell and spell-slot checks/deduction.
- Update failure text to: `No level 2 spell slots available. Use -i to bypass this check.`
- Include a line in successful output when `-i` was used, so the audit trail is explicit.
- Update `scribe` help text and tests.

Same issue elsewhere: this should be scoped to `scribe`; do not make `-i` silently bypass gold, downtime, materials, tool checks, or crafting checks unless explicitly designed.

## Issue 10: Crafting Help Examples Are For The Wrong Commands

Observed runtime help for `craft` shows examples:

- `!craft "Potion of Healing"`
- `!craft "Cloak of Protection"`

Investigation: these examples come from `crafting.help_text` in `src/gvars/utils/crafting/crafting.gvar`, not from the `.md` help file. The helper uses one generic example block for `craft`, `brew`, and `enchant`.

What went wrong: the generic helper ignores the actual command/catalogue split:

- `craft` is for mundane physical items/objects/vehicles.
- `brew` is for potions, tonics, poisons, and similar potion catalogue entries.
- `enchant` is for magic items.
- `scribe` is for spell scrolls.

Best fix:

- Give `craft`, `brew`, and `enchant` separate examples in `crafting.help_text`.
- Use only examples that resolve in the relevant configured catalogue.
- Update `.md` docs if they should mirror the runtime split more explicitly.

Same issue elsewhere: `brew` and `enchant` use the same generic example block, so all three command helps need to be corrected together.
