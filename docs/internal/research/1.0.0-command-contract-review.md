# 1.0.0 Command Contract Review

Status: pre-tag research.

`1.0.0` starts the public backwards-compatibility contract for command names, documented positional arguments, documented flags, and config keys used by those commands. After `1.0.0`, `1.x` releases should preserve these invocation shapes unless a change is purely additive or the old shape keeps working with a deprecation notice. Breaking command or config contract changes belong in `2.0.0`.

This review records the implemented command surface that should be deliberately accepted or changed before tagging `1.0.0`.

## Evidence Reviewed

- Alias inventory from `src/aliases/**/*.alias` and `utils/sourcemap.dev.json` / `utils/sourcemap.prod.json`.
- Source-adjacent alias help files in `src/aliases/**/*.md`.
- High-risk parsers for overloaded or stateful commands: `buy`, `sell`, `travel`, `loot`, `quest`, `recipe`, and `downtime`.
- Existing `.alias-test` coverage for command help, disabled-command behavior, common success paths, and representative edge cases.

No deployed `.alias` command is missing from the table below. The audit did find accepted synonym forms that were not explicit in the first draft of this review; those are now listed under the relevant commands and in the follow-up section.

## Summary

Recommendation: keep the current command names and argument layouts for `1.0.0`, subject to the pre-tag follow-ups below.

The command surface is internally consistent enough to lock as the first public contract. The areas with long-term compatibility risk are stateful or overloaded commands (`loot`, `buy`, `sell`, `travel`, `quest`, and `downtime`). Most high-risk paths have tests, but not every accepted synonym or optional form is directly tested.

Before tagging, any desired breaking simplification should happen now. After tagging, prefer additive aliases, extra flags, clearer validation, or deprecation messages rather than removing or repurposing existing tokens.

## Pre-Tag Decisions

No blocking command-layout changes are recommended from this review.

Sign off these compatibility commitments before tagging:

- `loot` keeps its stateful creature/item positional behavior.
- `loot reset` and `loot end` remain accepted clear-session aliases, or they are removed before `1.0.0`.
- `buy` keeps `yes` as the confirmation token and keeps the shop-first optional form.
- `sell` keeps the shop-first optional form.
- `travel` keeps `track`, `journey`, `next`, `reset yes`, and `set <location> yes`.
- `travel` currently ignores an unknown trailing token when an earlier prefix resolves to a destination, for example a route can still resolve before a non-transport word. Keep that behavior or tighten it before `1.0.0`.
- `downtime` keeps the accepted verb aliases even if docs emphasize `spend`.
- `quest` keeps `all`, `done`, `complete`, `completed`, `active`, `reopen`, and `activate` with their current list/status meanings.
- `recipe` keeps `ingredient`, `item`, and `uses` as ingredient-search tokens.

If any of those are undesirable as public API, change them before `1.0.0`; otherwise support them through the `1.x` line.

## Test Follow-Ups

These are not release blockers by themselves, but they are the coverage gaps found while reviewing the contract:

- Add direct `.alias-test` coverage for `!buy <shop> <item> [qty] [yes]`.
- Add direct `.alias-test` coverage for `!loot reset` and `!loot end`, or remove those synonyms before `1.0.0`.
- Add direct `.alias-test` coverage for `!quest all`, `!quest complete <quest>`, `!quest reopen <quest>`, and `!quest activate <quest>`.
- Add direct `.alias-test` coverage for `!recipe item <item>` and `!recipe uses <item>`.
- Decide whether the tested `!travel <location> <unknown-token>` fallback should remain compatible behavior or become an error before `1.0.0`.
